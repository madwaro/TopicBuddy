### 💬 Copilot

Here's a full breakdown of what this topic does:

---

## What This Topic Does

It's a **Copilot Studio AdaptiveDialog** — a conversational topic that lets an employee request time off through a chat interface, backed by Workday.

---

## How It Works (Step by Step)

### 1. **Automatic Input Extraction**
Before the conversation starts, the topic tries to silently extract from the user's message:
- `InputStartDate`, `InputEndDate` (dates mentioned)
- `InputHoursPerDay` (e.g., "4 hours a day")
- `InputTimeOffType` (e.g., "sick", "vacation", "PTO")

So *"Request sick leave next Monday"* can pre-fill the form.

### 2. **Keyword → Leave Type Mapping**
A config table maps fuzzy phrases to Workday dropdown values:
```
"vacation, pto, annual"  → Vacation_Hours
"floating, floater"      → Floating_Holiday_Hours
"sick, medical, illness" → Sick_Hours
```

### 3. **Fetch Leave Balances**
Calls the `msdyn_HRWorkdayHCMEmployeeGetVacationBalance` scenario, parses the response, and builds a display table of remaining hours per plan.

### 4. **Adaptive Card Form**
Shows the employee a card with:
- Their current balances (live from Workday)
- A dropdown for time off type (pre-selected if extracted)
- Start/end date pickers (pre-filled if mentioned)
- Hours per day input (defaults to 8)
- Submit / Cancel buttons

### 5. **Validation**
- If **Cancel** → politely ends
- If type = **"NOT_LISTED"** → redirects to Workday directly
- If **end date \< start date** → error + asks if they want to retry

### 6. **Build Date List & Submit**
Loops day-by-day from start → end date, builds a comma-separated list (e.g., `"2025-09-15,2025-09-16,2025-09-17"`), then calls the `msdyn_HRWorkdayAbsenceEnterTimeOff_MultiDay` API.

### 7. **Result**
- **Success** → Adaptive card confirmation showing type, date range, total days & hours
- **Failure** → AI rewrites the Workday error into a friendly message, then asks if they want to retry

---

## Key Design Patterns to Note
| Pattern | What it does |
|---|---|
| `AutomaticTaskInput` | Pre-fills form fields from the triggering message |
| `BeginDialog` + shared execution topic | Reusable wrapper for all Workday API calls |
| `AnswerQuestionWithAI` | Uses generative AI to humanize technical error messages |
| `GotoAction` | Loops the form back on validation failure or retry |
| Power Fx expressions inline | All logic (filtering, date math, string building) runs in Power Fx |

## 🔧 Configuring LSP Servers

GitHub Copilot CLI supports Language Server Protocol (LSP) for enhanced code intelligence. This feature provides intelligent code features like go-to-definition, hover information, and diagnostics.

### Installing Language Servers

Copilot CLI does not bundle LSP servers. You need to install them separately. For example, to set up TypeScript support:

```bash
npm install -g typescript-language-server
```

For other languages, install the corresponding LSP server and configure it following the same pattern shown below.

### Configuring LSP Servers

LSP servers are configured through a dedicated LSP configuration file. You can configure LSP servers at the user level or repository level:

**User-level configuration** (applies to all projects):
Edit `~/.copilot/lsp-config.json`

**Repository-level configuration** (applies to specific project):
Create `.github/lsp.json` in your repository root

Example configuration:

```json
{
  "lspServers": {
    "typescript": {
      "command": "typescript-language-server",
      "args": ["--stdio"],
      "fileExtensions": {
        ".ts": "typescript",
        ".tsx": "typescript"
      }
    }
  }
}
```

### Viewing LSP Server Status

Check configured LSP servers using the `/lsp` command in an interactive session, or view your configuration files directly.

For more information, see the [changelog](./changelog.md).
